# /copy-skill

GitHub上のスキルフォルダURLを受け取り、中身を取得して `~/.claude/skills/<name>/` に配置する。元が日本語ならそのままコピー、英語など外国語なら**内容を一切変えずに**日本語へ翻訳して配置する。

## 使い方

```
/copy-skill https://github.com/owner/repo/tree/main/path/to/skill
```

引数はGitHubの**フォルダURL**（`.../tree/<branch>/<path>` 形式）。そのフォルダ直下に `SKILL.md` がある前提。

## 全体フロー

```
URLパース → gh api で再帰取得 → 言語判定 → (翻訳 or そのまま) → 配置内容を要約提示 → 承認 → 書き込み → 最終サマリ
```

---

## 1. URLをパースする

入力URL `https://github.com/<owner>/<repo>/tree/<branch>/<path...>` を分解する。

- `owner`, `repo`, `branch`, `path` を取り出す。
- `tree` ではなく `blob`（単一ファイル）や、`/tree/<branch>` までしかない（リポジトリルート）場合は、その旨を伝えて中断する（このスキルは「SKILL.mdを含むフォルダURL」専用）。
- ブランチ名にスラッシュが含まれるケースがあるため、後続のContents API呼び出しで `?ref=<branch>` を明示する。判別が曖昧なときは `gh api repos/<owner>/<repo>` で `default_branch` を確認する。

## 2. gh api で中身を再帰取得する

`gh` は認証済み。Contents APIでフォルダ配下を取得する。

```bash
gh api "repos/<owner>/<repo>/contents/<path>?ref=<branch>"
```

- 返り値の各要素について:
  - `type == "dir"` → そのパスで再帰的に同じ呼び出しを行い、サブディレクトリ構造をそのまま再現する（例: `references/`）。
  - `type == "file"` → `download_url` から中身を取得する（テキストは `gh api .../contents/<file>?ref=<branch> --jq .content` をbase64デコードでも可）。
- フォルダ直下に `SKILL.md` が無ければ、「指定フォルダはスキルではない可能性」を伝えて中断する。
- 取得したファイル一覧とディレクトリ構造を手元に保持する。

## 3. 言語を判定する

`SKILL.md` の本文（自然言語部分）を読み、元の言語を判定する。

- **日本語** → 翻訳せず、全ファイルをそのままコピーする。
- **英語・その他の外国語** → 下記ルールで日本語へ翻訳する。

判定は本文を読んだ上での自然な判断でよい。言語判定の外部ツールは使わない。

## 4. 翻訳ルール（外国語の場合）

**大原則：内容を全く変えない。自然言語の散文だけを訳し、機械が解釈する識別子・コード・パスは1文字も変えない。**

### 訳す
- 本文の説明・手順・見出しなどの自然言語
- フロントマターの `description`（日本語で発動しやすくするため訳す。発動精度に関わる英語の重要キーワードは必要に応じて括弧で併記してよい）

### 原文のまま絶対に変えない
- フロントマターの `name`（機械可読ID。kebab-caseのまま）
- フロントマターの `trigger`（`/foo` 等のコマンド名）
- コードブロック・コマンド例・シェル/コードの中身
- **コードブロック内のコメントも原文維持**（コードには一切触れない）
- ファイル名・ディレクトリ名・パス（`references/foo.md` 等）
- URL、変数名、API名、フラグ（`--ja` 等）

### ファイル範囲
- **自然言語ファイル**（`.md`, `.txt` 等）→ すべて翻訳する。
- **コード・データファイル**（`.py`, `.sh`, `.js`, `.json`, 画像など）→ **バイト単位でそのまま保全**（翻訳も整形もしない）。

判定基準：「Claudeが読む自然言語か／機械が実行・解釈する資産か」。前者だけ訳し、後者は完全保全する。

## 5. 配置先と命名

- 配置先は固定で `~/.claude/skills/<name>/`。
- `<name>` は翻訳後も**元の `SKILL.md` フロントマターの `name` をそのまま使う**（GitHub上のフォルダ名ではなく `name` を正とする）。
- **同名のフォルダが既に存在する場合は、上書きせず中断してユーザーに確認する**（誤上書き防止）。

## 6. 出所の記録

配置する `SKILL.md` の**フロントマター直下に、コメントアウトで出所を残す**。フロントマター内のフィールドには追加しない（パース・発動挙動を汚さないため）。

```markdown
---
name: ...
description: ...
---
<!--
source: <元のGitHub URL>
imported_at: <YYYY-MM-DD>
-->
```

`imported_at` は実行日の日付を入れる。

## 7. 配置前の確認（承認ゲート）

ファイルを書き込む**前に**、以下を要約して提示し、ユーザーの承認を待つ。

- 配置先パス（`~/.claude/skills/<name>/`）
- 配置するファイル一覧とディレクトリ構造
- 元の言語と処理方針（例: 「英語 → 日本語訳」「日本語 → そのままコピー」）
- 取得物に**スクリプト（`.py`/`.sh` 等）が含まれる場合はその旨を明示**（中身は実行せず配置するだけだが、ユーザーが認識できるように）

承認後に書き込みを実行する。

## 8. 書き込みと最終サマリ

- 承認後、ディレクトリ構造を再現しながら全ファイルを配置する。
- 配置完了後、最終サマリを報告する:
  - 配置先パス
  - 翻訳したファイルと、原文のまま保全したファイルの内訳
  - 元URL（出所コメントに記録済みである旨）

## 注意・スコープ

- このスキルの責務は「外部スキルを翻訳・整形して手元に良い出発点として取り込む」まで. **カスタマイズはスコープ外**（取り込み後に通常どおり編集を依頼すればよい）。
- 取得したスクリプトは**実行しない**。配置するだけ。
- 入力は GitHub のフォルダURL のみ対応。単一ファイルrawリンク・リポジトリルート・任意Webページは対象外。
