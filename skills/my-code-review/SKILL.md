# My Code Review

diff の中から、設計品質の劣化を探す。自分の実装のセルフレビューと、checkout 済みの他人の PR のレビュー補助の両方で使う。

正しさのバグ・仕様適合・リポジトリの規約は対象外。それらは別のレビュー手段が担う。

検出は必ず**照合**（引用できる証拠）に基づく。設計の良し悪しの感想は書かない。

症状ラベル（Change Amplification・Information Leakage・Cognitive Load・Fake Generalization）の出典と定義は [GLOSSARY.md](GLOSSARY.md) を参照。

## Process

### 1. スコープを固定する

レビュー対象を diff に限定する。基準点（ブランチ・コミット）が未指定ならユーザーに確認する。`git diff <基準点>...HEAD` で差分を取得する。PR レビューの場合、対象 PR はローカルへ checkout 済みであることを前提とする（このスキル自身は gh などの外部ツールを呼ばない）。

完了条件：差分が空でないことを確認し、対象ファイルを列挙した。

### 2. 4つのチェックを全 hunk に適用する

各チェックは「発火 → 照合 → 処方」の順。**照合で証拠を引用できなければ、その候補は捨てる。**

#### Check 1: Implementation Leakage（重複コード）

- 発火：同じロジックの形が、diff 内の複数の hunk またはファイルに現れる。
- 照合：両方の箇所を引用できること。
- 処方：共通の形を抽出し、両方から呼ぶ。
- 症状ラベル：Change Amplification。

#### Check 2: Vocabulary Leakage（インターフェース語彙の出どころ照合）

- 発火：diff 内の公開メソッド名・引数名に、そのモジュールの層に属さない語が含まれる。両方向を見る。
  - 下向き：呼び手の文脈（画面名・機能名・ジョブ名・アクター名）が下位モジュールの名前に現れる。
  - 上向き：内部実装の詳細（テーブル名・アルゴリズム名・下位ライブラリの概念）が公開インターフェースに現れる。
- 照合：その語が実在する別レイヤーの成果物（クラス・画面・テーブル・ジョブ）と一致することをコードベースの検索で確認し、両方の出現箇所を引用できること。照合で裏取りできない「語彙センス」だけの指摘はしない。
- 処方：モジュール自身の操作の語彙にリネームする。ユースケースの知識は呼び手へ、実装の知識は内部へ。
- 症状ラベル：Information Leakage・Cognitive Load。

#### Check 3: One-Sided Argument（低確信度 — 本文と分けて「補足」に書く）

- 発火：diff で新規追加された nullable またはデフォルト値つき引数のうち、diff 内の呼び出し箇所で非デフォルト値を渡すのが1箇所だけのもの。
- 照合：diff 内の全呼び出し箇所を数えて列挙できること。
- 処方：その引数が表す関心事（監査・ログ・UI状態など）を別の仕組み（イベント・別呼び出し）に切り出す。
- 症状ラベル：Fake Generalization・Cognitive Load。

#### Check 4: Code-Repetition Comment（コメントがコードと同じ抽象レベルに留まっている）

- 発火：diff で新規追加または変更されたコメントが、直後のコード（変数宣言・式・呼び出し）が表す操作をそのまま言い換えているように見える。
- 照合：コメントとその対象コード行を引用できること。かつ、コメントから対象コードの識別子・演算子・呼び出し名にあたる語をすべて取り除いた後に残る語が、名詞（対象・理由・制約）ではなく動詞（操作の言い換え）だけであることを示せる。
- 処方：コメントを、対象（変数・引数・戻り値など）の「意味」や上位の意図を語る内容に書き換える。動詞（どう操作するか）ではなく名詞（それが何を表すか）に焦点を当てる。書き換えても情報が増えないなら、コメント自体を削除する。
- 症状ラベル：Change Amplification・Cognitive Load。

完了条件：4チェック × 全対象 hunk を適用し終え、すべての候補が「証拠引用つきで採用」か「捨てた」のどちらかになっている。

### 3. 報告する

- 最大3件、深刻度順。Check 1・2・4 は本文に、Check 3 は本文と分けて「補足」として書く。
- 各指摘は4点セット：チェック名＋症状ラベル＋hunk 引用＋処方。
- 発火したが証拠不足で捨てた候補があれば、末尾に「他 N 件を証拠不足で棄却」の一行を添える。
- 該当なしなら「漏洩は検出されなかった」の一行で終える。0件は正当な結果であり、埋め合わせの指摘をしない。

完了条件：報告した全指摘に引用が付いている。

## チェックを追加するときのルール

チェックの追加は、照合可能な証拠ルールを定義できるものに限る。以下は指摘してはいけないもの（負例）：

- 照合で裏取りできない命名の好み。
- リポジトリの規約や lint が既に扱っている事項。規約は本スキルより常に優先する。
